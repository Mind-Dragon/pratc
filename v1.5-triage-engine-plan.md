# prATC v1.5 — Triage Engine That Actually Works

## Release Goal

v1.4.2 can sync and analyze a 6,646-PR repo. But the output is noise:
- 92,911 "conflicts" (file overlap, not merge conflicts)
- 0 duplicate groups (detector broken by quick-skip threshold)
- No garbage/junk layer (every PR gets reviewed equally)
- Review pipeline puts 95% of PRs into "needs_review" with no differentiation
- The PDF report doesn't answer "what is good and what is garbage"

v1.5 makes the triage engine produce actionable output: a report that tells a human what to merge, what to close, and what to ignore, with clear reasons.

## What's Broken (from openclaw/openclaw production run)

### Issue 1: Duplicate detection returns 0 groups

**Symptom:** 6,646 PRs, 0 duplicate groups, 68 overlap groups.
**Cause:** The quick-skip threshold (`titleScore < 0.1`) is too aggressive. A PR pair with titles "fix: update deps" and "chore: bump dependencies" has a low Jaccard title score but is clearly a duplicate. The body contributes 10% to the score, which isn't enough to pull it over the 0.90 duplicate threshold.
**Fix:**
- Lower the quick-skip threshold from 0.1 to 0.05 (or remove it entirely for force-cache mode since we have time)
- Weight body similarity higher (currently 10%, needs to be 20-30%)
- Add file-overlap as a primary signal: if two PRs touch the same files AND have ANY title similarity, flag as potential duplicate
- Add a "near-duplicate" threshold between 0.70-0.90 for borderline cases

### Issue 2: Conflict count is useless (92,911)

**Symptom:** 92,911 conflict pairs in a 6,646-PR repo. That's ~14 conflicts per PR on average.
**Cause:** `buildConflicts()` checks if any two PRs share a file. In a monorepo, this produces massive noise — every PR touches `package.json`, `pnpm-lock.yaml`, shared config files.
**Fix:**
- Filter out "noise" files: `package.json`, `pnpm-lock.yaml`, lock files, `.github/` configs, `.eslintrc`, `tsconfig.json`, `.gitignore`, CI config files
- Only count conflicts on "signal" files: source code, tests, docs, config schemas
- Add a "conflict severity" score based on:
  - Number of conflicting files (1 shared file = weak, 5+ shared files = strong)
  - Whether files are source code vs config
  - Whether the files are in the same subsystem/directory
- Collapse conflicts into "merge-blocking" (can't merge together) vs "attention-needed" (might conflict, review required)

### Issue 3: No garbage/junk classifier

**Symptom:** Every PR gets reviewed by the review pipeline. There's no pre-filter to remove obviously bad PRs before deep analysis.
**Cause:** The "outer peel" phase (layers 1-3 from GUIDELINE.md) is not implemented as a standalone pass.
**Fix — implement a `garbage` pass before `classifyDuplicates`:**
- **Abandoned:** no activity (no commits, no comments) in > 90 days AND author has no other recent PRs
- **Bot PRs:** already detected by `internal/analysis/` but not used to filter
- **Empty PRs:** 0 additions, 0 deletions, 0 changed files (or just metadata changes)
- **Spam patterns:** title/body matches known spam signatures (e.g., "test", "asdf", single-character titles, URLs only)
- **Malformed:** missing title, missing body AND missing files, or git error metadata
- **Closed/merged PRs that are still "open" in cache:** mark as stale with reason
- Each garbage PR gets assigned to the `junk` bucket with a reason code

### Issue 4: Review pipeline doesn't differentiate

**Symptom:** 95% of PRs end up in "needs_review" or "merge_after_focused_review" with no differentiation.
**Cause:** The review analyzers (security, reliability, quality) produce findings, but the scoring doesn't aggregate them into a meaningful substance score. Every PR that passes the outer peel gets the same treatment.
**Fix:**
- Add a `substance_score` (0-100) computed from:
  - File change count and depth (touching many files in core subsystems = higher substance)
  - Test coverage delta (PRs that add/update tests score higher)
  - Review analyzer findings (security/risk findings lower score, quality findings raise it)
  - Author history (first-time contributors get a review flag, not a penalty)
  - PR age (fresh PRs score higher than 6-month-old PRs with no activity)
- Use the substance score to differentiate:
  - `high_value` (score >= 70)
  - `needs_review` (score 30-69)
  - `low_value` (score < 30)
  - Route `high_value` to `now` temporal bucket
  - Route `low_value` to `future` temporal bucket

### Issue 5: Deep judgment layers (6-16) not implemented

**Symptom:** The GUIDELINE.md describes 16 layers but only layers 4-5 exist in any form. Layers 6-16 are documented but not computed.
**Cause:** The deep judgment layers were planned but never implemented.
**Fix — implement as scores that refine the bucket assignment:**

| Layer | Field | Data Source | Implementation |
|-------|-------|------------|----------------|
| 6. Confidence | `confidence` (0-1.0) | Count of review findings with confidence > 0.7 / total findings | If < 0.5 findings, mark as `needs_review` |
| 7. Dependency | `blocked_by` []int | Conflict graph: PRs that must merge before this one | Mark as `blocked` if any blocker exists |
| 8. Blast radius | `blast_radius` (low/med/high) | Number of downstream files affected + subsystem criticality | High = `needs_review` with security flag |
| 9. Leverage | `leverage` (0-1.0) | Number of PRs this unblocks / total blocked PRs | High leverage = `high_value` |
| 10. Ownership | `has_owner` (bool) | Author is active (has commits in last 30 days) AND PR is not abandoned | No owner = `re_engage` or `stale` |
| 11. Stability | `staleness_signal` | From existing staleness analysis | Already partially implemented |
| 12. Mergeability | `mergeable` | From GitHub API `mergeable` field | Already captured but not used for routing |
| 13. Strategic weight | `strategic` (0-1.0) | Subsystem match to known project priorities (from settings) | Configurable per-repo |
| 14. Attention cost | `attention_cost` (low/med/high) | File count + diff size + number of review findings | High cost = defer to `future` |
| 15. Reversibility | `reversible` (bool) | Touches only test files, docs, or config? | Reversible = lower risk |
| 16. Signal quality | `signal_quality` (noise/signal) | Composite of layers 6-15 | Final gate before human review |

### Issue 6: PDF report is not useful

**Symptom:** The report exists but doesn't answer "what is good and what is garbage."
**Cause:** The report renders raw data without the outer peel applied first.
**Fix:**
- Apply the garbage pass before generating the report
- Section order in the PDF must match the GUIDELINE.md decision map:
  1. Executive summary (total PRs, breakdown by bucket)
  2. Junk/noise section (garbage PRs with reasons)
  3. Duplicates section (canonicals + chains)
  4. "Do this now" (high_value + merge_candidate)
  5. "Needs review" (needs_review with risk flags)
  6. "Defer to future" (low_value, future, blocked)
  7. Full appendix (every PR with bucket + reason codes)
- The executive summary should say something like: "6,646 PRs analyzed. 234 are garbage, 187 are duplicates of existing work. 142 are high-value merge candidates. 6,083 need human review."

## Work Items (in dependency order)

### Phase 1: Outer Peel (Layers 1-3)

**1.1 Garbage classifier pass** (`internal/app/service.go`)
- New function `classifyGarbage(prs []types.PR) []types.PR` that returns filtered PRs with garbage reasons
- Run before clusters and duplicates
- Assign `junk` bucket with reason codes
- Emit `garbage` progress phase
- Estimate: 50% of 6,646 PRs are garbage in a repo like openclaw/openclaw

**1.2 Fix duplicate detection** (`internal/app/service.go`)
- Lower quick-skip threshold from 0.1 to 0.03
- Add file-overlap as a primary signal
- Weight body similarity at 20% instead of 10%
- Add "near-duplicate" bucket (0.70-0.90 overlap)
- Keep progress emits

**1.3 Fix conflict graph** (`internal/app/app.go`, `internal/app/service.go`)
- Filter noise files from conflict detection
- Add conflict severity scoring
- Separate "merge-blocking" conflicts from "attention-needed"
- Reduce from 92,911 noise to actionable conflict count

### Phase 2: Substance Scoring (Layer 4-5)

**2.1 Substance score calculation**
- New function `computeSubstanceScore(pr types.PR, analyzers []ReviewResult) int`
- Use file depth, test coverage delta, review findings, author activity, PR age
- Wire into the review pipeline

**2.2 Now vs future routing**
- Use substance score + staleness + mergeability to assign temporal bucket
- `high_value` + `mergeable` + not stale → `now`
- `high_value` + not mergeable or stale → `future`
- `low_value` → `future`
- `blocked` → `blocked`

### Phase 3: Deep Judgment (Layers 6-16)

**3.1 Add fields to types.AnalysisResponse**
- `Confidence float64`
- `BlockedBy []int`
- `BlastRadius string` (low/med/high)
- `Leverage float64`
- `HasOwner bool`
- `StrategicWeight float64`
- `AttentionCost string` (low/med/high)
- `Reversible bool`
- `SignalQuality string` (noise/signal)

**3.2 Compute each layer**
- Layer 6-8: from review findings + conflict graph
- Layer 9-10: from author activity + dependency graph
- Layer 11-12: from staleness + GitHub mergeable field
- Layer 13-14: from settings + file count
- Layer 15-16: from composite of other layers

**3.3 Use layers to refine bucket assignment**
- Apply the layer scores as final gates before human review
- A PR that passes all layers without flags goes to `merge_candidate`
- A PR that fails multiple layers goes to `needs_review` with specific flags

### Phase 4: Report Composition

**4.1 Restructure PDF sections**
- Executive summary with bucket counts
- Junk section with reason codes
- Duplicate chains
- Now queue (sorted by substance score)
- Needs review (sorted by risk)
- Future/blocked
- Full appendix

**4.2 Add recommendations section**
- "Close these N garbage PRs"
- "These N duplicates have a canonical — close the rest"
- "These N PRs are high-value and mergeable — merge them now"
- "These N PRs need focused review before merge"

### Phase 5: Performance

**5.1 Parallelize duplicate detection**
- Use goroutine workers for the outer loop
- Each worker handles a chunk of outer iterations
- Results merged at the end
- Target: < 30s for 6,000+ PRs

**5.2 Cache intermediate results**
- Store duplicate groups, conflict graph, substance scores in SQLite
- Reuse on subsequent runs instead of recomputing
- Only recompute when the PR set changes
