# prATC v1.7 Workstream 1 — Diff Analysis Implementation Plan

> For Hermes: execute this in small commits with a failing contract test before each behavior change. Keep all additions additive-safe.

**Goal:** Ship real diff-grounded evidence for v1.7 by adding subsystem tagging, risky-pattern detection, and bounded evidence surfacing to JSON and PDF outputs.

**Architecture:** Reuse the existing diff path already wired through `internal/repo/mirror.go` → `internal/app/service.go` → `types.PRAnalysisData`. Keep the blast radius low by extending `AnalyzerFinding` production in analyzers and by surfacing those findings through existing `ReviewResult.AnalyzerFindings` and report rendering.

**Tech stack:** Go, stdlib, existing review analyzers, existing PDF/report pipeline.

---

## What is already true in code

### Live diff path
- `internal/repo/mirror.go:GetDiffPatch()` fetches git diff patches from the mirror
- `internal/repo/mirror.go:parseDiffOutput()` parses patch content into `[]types.PRFile` and `[]types.DiffHunk`
- `internal/app/service.go` conditionally populates `prData.Files` and `prData.DiffHunks`
- `internal/review/analyzer_security.go` already uses real diff-grounded evidence

### Existing output structures we should reuse
- `internal/types/models.go:ReviewResult.AnalyzerFindings`
- `internal/types/models.go:AnalyzerFinding`
- `internal/types/models.go:PRFile`
- `internal/types/models.go:DiffHunk`

### Current gaps
- Only the security analyzer uses diff-grounded evidence today
- No explicit subsystem tagging model exists
- Quality/reliability/performance analyzers do not inspect patch content
- Analyst/PDF surfaces do not clearly expose `AnalyzerFindings`

### Task 3: Create deterministic subsystem classifier

**Objective:** Classify changed files into stable subsystem tags using path rules.

**Files:**
- Create: `internal/review/diff_subsystems.go`
- Create: `internal/review/diff_subsystems_test.go`

**Initial subsystem tags:**
- `security`
- `auth`
- `api`
- `database`
- `config`
- `infra`
- `tests`
- `docs`
- `frontend`

**Rules:**

**Steps:**

### Task 4: Add risky-pattern detectors over patch content

**Objective:** Detect real risky changes from diff hunks and file patches.

**Files:**
- Create: `internal/review/diff_patterns.go`
- Create: `internal/review/diff_patterns_test.go`

**Initial risky pattern classes:**
- auth checks / permission logic
- token / secret / credential handling
- SQL query construction / migration-sensitive changes
- crypto operations
- production-code edits without adjacent tests (signal only; full test evidence is WS3)

**Rules:**
- Use clear regex/string heuristics
- Findings must include `AnalyzerName`, `Finding`, `Confidence`, `Location`, `EvidenceHash`
- Use `SignalType` and `Subsystem` to distinguish the nature of the evidence
- Keep snippets bounded and deterministic

**Steps:**
1. Write focused failing tests per risky pattern.
2. Implement pure helpers operating on `[]types.PRFile` and `[]types.DiffHunk`.
3. Keep them independent from orchestration code.
4. Run tests.

**Commands:**
- `go test ./internal/review/... -run TestDetect.*Diff -v`

**Commit:** `feat(ws1): add diff risky-pattern detectors`

## Recommended first implementation cut

Start with Tasks 1-3 only.

That gives us:
- proof the live diff path is real
- an explicit subsystem model
- deterministic subsystem classification

It keeps blast radius small before touching analyzer behavior.

## Non-goals for WS1

- No probabilistic classification
- No ML-based code understanding
- No semantic AST diff engine
- No cache schema expansion for full diff persistence yet
- No broad PDF redesign

## Exit condition

WS1 is complete when prATC can say, for a reviewed PR, not just "security analyzer fired" but:
- which subsystem changed
- what risky pattern was seen in the diff
- where the evidence came from
- and that evidence appears in the JSON/report surfaces without breaking existing clients
