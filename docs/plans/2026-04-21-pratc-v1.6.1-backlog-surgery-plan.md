# prATC v1.6.1 — Backlog Surgery Plan

> For Hermes: implement this plan locally, keep the repo on local-only changes unless the user explicitly asks for remote actions.

**Goal:** Expand the merge plan surface beyond the current top-20 ceiling by recursively collapsing duplicate groups, recovering viable blocked PRs, and re-ranking low_value PRs. Turn the 6,632-PR corpus into a larger actionable merge queue.

**Picks up from:** INTERNAL-ROADMAP.md #4 (Backlog surgery / canonicalization) and the live v1.6 openclaw run that surfaced 3,334 blocked + 2,824 low_value PRs with only 20 selected.

**Architecture:** Keep `internal/app/service.go` as the pipeline orchestrator. Add new post-review passes: `duplicate_collapse.go`, `blocked_recovery.go`, `low_value_rerank.go`. The planner (`internal/planner/`) gets a collapse-aware pre-pass. The report (`internal/report/`) gets new sections for recovery queues and batch recommendations.

---

## Slice 1 — Recursive Duplicate Expansion

### Objective
Currently the planner sees 6,632 PRs and selects 20. Duplicate detection finds 95 groups, but they still compete for plan slots as individual PRs. Collapse each duplicate group to its canonical representative *before* planning, then re-run the planner on the collapsed corpus. This should free up plan capacity for non-duplicate PRs.

### Implementation paths
- `internal/app/duplicate_synthesis.go` — existing canonical nomination logic
- `internal/planner/pool.go` or `internal/planner/selector.go` — plan candidate pool construction
- `internal/app/service.go` — orchestrate collapse before planning
- `internal/report/analyst_sections.go` — new "Duplicate Collapse Impact" section

### Required changes
1. **Collapse pass**
   - After `buildDuplicateSynthesis()`, produce a `CollapsedCorpus` map: `canonicalPR → []supersededPRs`
   - Replace all superseded PRs in the planning pool with their canonical representative
   - Mark canonical PRs with `IsCollapsedCanonical = true` and attach `SupersededPRs` list
2. **Recursive collapse**
   - A canonical PR may itself be in another duplicate group (chain). Detect and flatten chains.
   - Ensure no PR is both canonical and superseded in the final collapsed corpus.
3. **Planner integration**
   - Add `plan --collapse-duplicates` flag (default: true in v1.6.1)
   - The planner runs on `len(PRs) - len(superseded)` candidates
   - Target should auto-adjust: if corpus shrinks by N, target can grow proportionally
4. **Report surface**
   - New PDF section: "Duplicate Collapse Impact" showing:
     - Original PR count → Collapsed PR count
     - Groups collapsed
     - Plan slots freed
     - Top canonical PRs nominated

### Acceptance checks
- Collapsing 95 duplicate groups in openclaw reduces planning pool by ≥90 PRs
- No duplicate PR appears in the final merge plan (only canonicals)
- Chain duplicates are flattened (A→B→C becomes A→[B,C])
- PDF shows collapse impact summary

## Slice 2 — Automatic Second Pass: Blocked PR Reclassification

### Objective
3,334 blocked PRs (50.3%) exit the first pass with `Category = blocked` and are never reconsidered. Add an **automatic second pass** that re-evaluates blocked PRs after the main pipeline. If a path forward exists, reclassify the PR into a viable bucket (needs_review, high_value, or merge_candidate). If permanently stuck, downgrade to junk or keep as blocked with a clear reason.

### Why automatic matters
The user does not want a one-off recovery report. They want the pipeline itself to find paths forward. A blocked PR that is actually `draft + active author + light conflicts` should leave the blocked bucket and enter the planning pool automatically.

### Implementation paths
- `internal/review/orchestrator.go` — add `RunSecondPass()` after main review
- `internal/review/recovery.go` — NEW: automatic reclassifier for blocked PRs
- `internal/types/models.go` — add `ReclassifiedFrom` and `ReclassificationReason` fields
- `internal/report/analyst_sections.go` — show reclassified PRs in Decision Trail

### Required changes
1. **Second pass architecture**
   - After `buildDecisionLayers()` completes for all PRs, collect PRs where `Category == blocked`
   - Run `ReclassifyBlocked(prData, firstPassResult)` for each
   - The function returns a new `ReviewResult` with updated fields:
     - `Category` — may change to needs_review, high_value, merge_candidate, or junk
     - `DecisionLayers` — append a recovery layer (Gate 17: Recovery Assessment)
     - `ReclassifiedFrom` — original category (`blocked`)
     - `ReclassificationReason` — human-readable path forward
     - `NextAction` — updated action
   - If no path forward, keep `Category = blocked` but add a `permanent_blocker` reason

2. **Recovery rules** (deterministic, auditable)
   | Condition | New Category | Reason |
   |-----------|--------------|--------|
   | CI failing + last push <30d + not draft + <3 conflicts | needs_review | Author is active; likely fixable CI issue |
   | Draft + ≥2 reviews/activity + not stale | high_value | Close to ready; just needs publish |
   | Mergeable=unknown + small size + no risk flags | needs_review | Needs human check but not blocked |
   | CI failing + last push >90d + no reviews | junk | Abandoned; recommend close |
   | >10 conflict pairs + stale | junk | Too entangled; not worth recovery |
   | Spam/junk markers in title/body | junk | Confirmed low-quality |
   | All others | blocked | Permanent blocker with explicit reason |

3. **Planner integration**
   - The planner sees the updated categories automatically
   - No new CLI flags needed — this is part of the standard pipeline
   - The report shows reclassified PRs with a → arrow notation

4. **Report surface**
   - Decision Trail marks reclassified PRs: `#5649 [blocked → needs_review]`
   - New PDF subsection: "Reclassified from Blocked" showing count and top examples

### Acceptance checks
- Second pass runs in <5 seconds on 3,334 blocked PRs
- ≥10% of blocked PRs reclassified to viable buckets on openclaw
- Zero spam/junk PRs promoted to viable buckets
- Report shows reclassification arrows and reasons
- All reclassifications are deterministic and reproducible

## Slice 3 — Automatic Second Pass: Low_Value PR Reclassification

### Objective
2,824 low_value PRs (42.6%) are deprioritized in the first pass. Add an **automatic second pass** that re-evaluates low_value PRs. If a PR is actually a quick win, safe batch-merge candidate, or hidden high-value work, promote it. If it is stale garbage, downgrade to junk.

### Why automatic matters
Same as Slice 2 — the pipeline itself should find hidden value, not just report on it. A docs PR with CI green and no conflicts is often a `quick_win`, not a `low_value` burden.

### Implementation paths
- `internal/review/orchestrator.go` — extend `RunSecondPass()` to also handle low_value
- `internal/review/quickwin.go` — NEW: automatic reclassifier for low_value PRs
- `internal/types/models.go` — reuse `ReclassifiedFrom` / `ReclassificationReason`
- `internal/report/analyst_sections.go` — show reclassified low_value PRs

### Required changes
1. **Second pass architecture**
   - After the blocked recovery pass, collect PRs where `Category == low_value`
   - Run `ReclassifyLowValue(prData, firstPassResult)` for each
   - Returns updated `ReviewResult`:
     - `Category` — may change to merge_candidate, high_value, needs_review, or junk
     - `DecisionLayers` — append a rerank layer (Gate 17: Value Reassessment)
     - `ReclassifiedFrom` — `low_value`
     - `ReclassificationReason` — why it was promoted or demoted

2. **Re-rank rules** (deterministic, auditable)
   | Condition | New Category | Reason |
   |-----------|--------------|--------|
   | size_XS/S + CI passing + not draft + <3 conflicts + substance >30 | merge_candidate | Safe quick win; batch-mergeable |
   | Docs cluster + CI passing + no conflicts + substance >25 | high_value | Docs are valuable; easy merge |
   | Security/reliability/perf findings + substance >40 | needs_review | Hidden risk/value needs review |
   | No activity >90d + CI failing + no reviews + substance <20 | junk | Stale and worthless |
   | All others | low_value | Genuine low_value; keep in defer queue |

3. **Batch-merge tagging**
   - For PRs reclassified to `merge_candidate` from `low_value`, add a `BatchTag` field
   - Tags are derived from cluster or file patterns: `docs-batch`, `typo-batch`, `dependency-batch`
   - The planner can optionally batch-select these

4. **Report surface**
   - Decision Trail marks reclassified PRs: `#10195 [low_value → merge_candidate]`
   - New PDF subsection: "Reclassified from Low Value" showing count and top examples
   - Batch tags shown where applicable

### Acceptance checks
- ≥5% of low_value PRs reclassified to viable buckets on openclaw
- Zero high-risk PRs promoted to merge_candidate
- Batch tags make sense (e.g., docs PRs cluster together)
- Report shows reclassification arrows, reasons, and batch tags
- All reclassifications are deterministic and reproducible

## Slice 4 — Expanded Merge Plan Target

### Objective
The current planner targets 20 PRs and finds exactly 20 candidates. Make the target dynamic so the plan scales with corpus health.

### Implementation paths
- `internal/planner/selector.go` — target calculation
- `internal/cmd/plan.go` — CLI flags
- `internal/report/pdf.go` — plan summary

### Required changes
1. **Dynamic target calculation**
   - Base target: `max(20, corpus_size * 0.003)` (0.3% of corpus, min 20)
   - For openclaw: 6,632 * 0.003 = ~20 (same as now)
   - With duplicates collapsed: pool shrinks, so target stays ~20
   - With recoverable + quick_wins added: pool grows, target should expand
   - Formula: `target = max(20, viable_pool * 0.05)` where viable_pool = non-junk, non-abandoned PRs
2. **CLI flags**
   - `--target-ratio float` — target as % of viable pool (default: 0.05 = 5%)
   - `--min-target int` — minimum target (default: 20)
   - `--max-target int` — maximum target (default: 100)
3. **Planner behavior**
   - If viable_pool < min_target, use min_target
   - If calculated target > max_target, cap at max_target
   - Report shows: "Target: X PRs (Y% of viable pool Z)"

### Acceptance checks
- Default target on openclaw with v1.6.1 features is >20
- `--target-ratio=0.10` produces target ~50-60 on openclaw
- `--min-target=50` forces at least 50 candidates
- Planner does not degrade in runtime when target increases
