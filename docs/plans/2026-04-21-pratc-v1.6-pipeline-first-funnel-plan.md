# prATC v1.6 Pipeline-First Funnel Plan

> For Hermes: implement this plan locally, keep the repo on local-only changes unless the user explicitly asks for remote actions.

Goal: reset prATC around CLI + API + PDF, strengthen the 16-gate funnel into a mandatory ordered pipeline, add first diff-grounded evidence, and design duplicate-group synthesis outputs for a future merge bot.

Architecture: keep `internal/app/service.go` as the top-level pipeline orchestrator, `internal/review/` as the gate engine, `internal/cmd/serve.go` as the AI-facing API surface, and `internal/report/` as the human-facing PDF surface. Remove or quarantine browser/dashboard assumptions so the system reads as a long-running machine pipeline with a PDF packet, not a UI product.

Tech stack: Go service + CLI, SQLite cache/state, PDF generation in Go, optional ML bridge only where it helps pipeline stages, JSON API for AI consumers.

## Slice 2 — Make the 16-gate ladder a real funnel contract

### Objective
Convert the existing 16-layer review output from descriptive metadata into an explicit mandatory gate journey shared by every PR.

### Key implementation paths
- `GUIDELINE.md`
- `ARCHITECTURE.md`
- `internal/app/service.go`
- `internal/review/orchestrator.go`
- `internal/types/models.go`
- `internal/report/`

### Required changes
1. Introduce an explicit gate model.
   - Each gate needs: gate number, gate name, cost tier, input status, outcome, reasons, exit/advance decision.
2. Make every PR start at Gate 1.
3. Encode early-exit semantics.
   - Junk, duplicate, or obvious-bad PRs exit early but remain visible.
4. Preserve the fixed order.
   - No skipping straight to later gates without explicit earlier outcomes.
5. Distinguish gate classes.
   - elimination gates
   - scoring gates
   - routing gates
   - deep-judgment gates
6. Expose the full journey in API and PDF.

### Suggested type additions
Consider extending or replacing `DecisionLayer` with a more explicit gate record:
- `gate`
- `name`
- `cost_tier`
- `status`
- `outcome_bucket`
- `reasons`
- `continued`
- `terminal`

### Acceptance checks
- Every analyzed PR has a full ordered gate trail
- Early exits are explicit, not implied
- no PR gets category/temporal bucket without earlier gate outputs
- focused tests under `internal/review/` and `internal/app/` cover ordered gate progression

## Slice 3 — Add first diff-grounded evidence to v1.6

### Objective
Push the funnel beyond metadata heuristics by introducing a narrow but real diff-aware evidence layer.

### Key implementation paths
- `internal/github/` or `internal/repo/` diff fetch/cache paths
- `internal/types/models.go`
- `internal/review/analyzer_security.go`
- `internal/review/analyzer_reliability.go`
- `internal/review/analyzer_performance.go`
- `internal/review/analyzer_quality.go`
- `internal/review/orchestrator.go`
- `internal/report/`

### Scope for v1.6
Keep this narrow and high value.

Start with evidence classes like:
- auth/security checks touched in diff hunks
- risky config changes in diff hunks
- test-gap evidence: production code changed without test evidence
- performance-sensitive query/cache/loop signatures in changed hunks

### Rules
- Outer gates stay cheap and metadata-based.
- Diff-aware work only runs on PRs that survive the outer peel.
- Evidence must be attachable to findings with location/snippet context.

### Output expectation
A review finding should evolve from:
- "auth file changed"

to something closer to:
- "auth guard logic changed in `path/file.go` near line X; no adjacent test evidence found"

### Acceptance checks
- at least one analyzer emits code-location-backed findings
- API response includes concrete evidence references
- PDF includes diff-grounded reasons for the highest-signal findings
- large-corpus cost remains bounded because diff analysis is not run on obvious trash

## Slice 4 — Duplicate groups to synthesis-ready outputs

### Objective
Advance duplicate handling from detection to canonical nomination and future merge-by-bot planning.

### Key implementation paths
- `internal/app/service.go`
- `internal/types/models.go`
- `internal/review/orchestrator.go`
- `internal/report/`
- `docs/api-contracts.md`

### Required changes
1. Extend duplicate groups with richer ranking inputs.
   - completeness
   - freshness
   - mergeability
   - test evidence
   - conflict footprint
   - confidence / signal quality
2. Nominate a canonical PR explicitly.
3. Mark alternates and salvageable contributors.
4. Define a synthesis artifact for future bot consumption.

### Synthesis artifact idea
For each near-duplicate group, emit something like:
- canonical candidate
- supporting PRs
- rejected PRs
- strongest evidence found in each PR
- conflicting parts
- recommended merge/synthesis order
- explicit reasons for selection

### Important v1.6 limit
Do not mutate GitHub.
This slice only produces a machine-readable recommendation packet for a future bot.

### Acceptance checks
- duplicate groups have a nominated canonical
- near-duplicate groups can emit a synthesis recommendation object
- report surfaces show canonical + alternates clearly
- API makes the future bot contract obvious and stable
